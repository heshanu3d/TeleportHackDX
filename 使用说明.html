TeleportHackDX 使用说明 DirectX9 + ImGui 内嵌界面版 TeleportHack — 构建、注入与使用全流程

1. 这是什么

TeleportHackDX 是 TeleportHackOnVanilla（原 AutoIt 版本）的完整 C++/ImGui 移植版。 它不再是一个外部 AutoIt/Python 进程通过 ReadProcessMemory 远程读写 WoW 内存， 而是一个被注入进 WoW.exe 进程内部的 DLL：它 Hook 住 Direct3D9 的 EndScene / Reset，直接在游戏画面上绘制一个 Dear ImGui 界面， 和 AzDeltaQQ/WoW3.3.5aDirectXHookWithGui 参考项目使用相同的 Hook 方式。

支持的客户端版本：1.12.1（Vanilla）、1.12.3（Vanilla）、3.3.5a（WLK），均为 32 位客户端。

2. 准备工作（需要一台 Windows 机器）

这个 DLL 必须运行在真实的 Windows 进程内部（Hook DirectX9），无法在 Linux/Ubuntu 上运行， 也不建议用 Wine（Wine 下 DirectX9 Hook 和游戏客户端本身兼容性都不可靠）。你需要：
  • 一台 Windows 10/11 物理机，或者
  • 一台 Windows 虚拟机（VMware / VirtualBox / 云端 Windows VM 均可）
在这台机器上安装好目标版本的 WoW 客户端（32 位）。

3. 获取产物

方式 A：直接下载 CI 构建产物（推荐，无需装 Visual Studio）

  1. 打开仓库 Actions 页面： https://github.com/heshanu3d/TeleportHackDX/actions
  2. 点击最新一次状态为绿色 ✅ 的 build workflow run
  3. 页面下方 Artifacts 区域下载 TeleportHackDX-win32.zip
  4. 解压后得到：TeleportHackDX.dll、favlist.fav、hotkey.txt

方式 B：本地用 Visual Studio 编译

git clone git@github.com:heshanu3d/TeleportHackDX.git
cd TeleportHackDX
mkdir build && cd build
cmake -A Win32 ..
cmake --build . --config Release
# 产物在 build\bin\Release\TeleportHackDX.dll

需要 Visual Studio 2019+ 和 CMake 3.15+；必须编译为 32 位（Win32）， 因为 WoW 这几个版本客户端都是 32 位程序。不需要额外安装老版 DirectX SDK。

4. 整理文件目录

把下面这些文件放在同一个文件夹里（这个文件夹可以在磁盘任意位置， 不需要和 WoW.exe 放一起）： 

C:\WowHack\
├── TeleportHackDX.dll
├── favlist.fav
└── hotkey.txt
为什么必须放一起？因为这个 DLL 被注入进游戏进程后，没有常规意义上的"当前工作目录"。 代码里（src/util/paths.cpp）会在运行时反查"自己这个 DLL 文件在磁盘上的哪个目录"， 然后固定去这个目录下寻找 favlist.fav / hotkey.txt / settings.json。 也就是说这三个文件跟着 DLL 走，跟 WoW 装在哪里没有关系。

你的注入工具放这里

仓库里预留了 TeleportHackDX/dll_injector/ 目录，你可以把下载好的注入工具 （见下一节）放进去，方便和这个项目放在一起管理。

5. 注入到 WoW.exe

  1. 1启动 WoW 客户端（对应版本，32 位），进到角色选择界面或者进入游戏世界都可以 ——只要窗口和 D3D9 设备已经创建即可。
  2. 2选一个通用 DLL 注入工具（任选其一，放进 dll_injector/ 目录）：
    工具特点操作方式
    System Informer
    （原 Process Hacker，推荐）    		 免费开源，界面直观，日常最常用的进程工具之一 右键 WoW.exe 进程 → Miscellaneous → Inject DLL... → 选择 TeleportHackDX.dll
    Extreme Injector 专用注入器，界面简单，新手友好 选中 WoW 进程 → 浏览选择 DLL → 注入方式保持默认 LoadLibrary → 点击 Inject
    Xenos 功能较全的注入器 操作流程类似 Extreme Injector
    Cheat Engine 自带的注入功能也可以用，但它 README 里提到的驱动改名技巧是针对 带反作弊驱动的正版游戏的，这几个怀旧服/私服客户端一般不需要。
  3. 3注入成功后，游戏内按 Win + Insert 显示/隐藏悬浮界面。

6. 界面使用说明

界面布局完全对应原 AutoIt/Python 版本的窗口结构：

区域说明
client 下拉框选择当前客户端版本：1.12.1 / 1.12.3 / 3.3.5，默认 3.3.5，务必先选对再传送
分类 / 搜索按描述前缀（-之前的部分）分类筛选，搜索框按描述模糊匹配
传送点表格显示 描述/x/y/z；单击选中一行，双击直接传送
描述输入框新增/编辑传送点时填写描述文本
step 勾选框勾选后传送变为"分步平滑移动"，不勾选则瞬间传送
AntiJump / AutoLoot / LuaUnlock / PatchLoot四个内存补丁开关按钮，点一次切换一次状态，结果显示在下方日志里
插前 / 插后 / 编辑 / 删除基于当前选中行 + 描述输入框 + 当前读取到的角色坐标，插入/编辑/删除该分类下的传送点
追加 / 保存 / 重载追加=在列表末尾新增一条；保存=写回 favlist.fav；重载=从磁盘重新读取 favlist.fav + hotkey.txt
Teleport 大按钮传送到当前选中的传送点
Speed 行设置移动速度（仅 3.3.5 支持，1.12.x 灰色不可用）
Import / Export导入替换全部 / 合并去重导入 / 导出到指定 .fav 文件（弹出系统文件选择框）
Settings调整分步移动的步距/间隔、favlist.fav 和 hotkey.txt 的路径
日志区所有操作结果、错误信息都会打印在这里

另外，hotkey.txt 里配置的组合键（如 ^1 = Ctrl+1）无论悬浮界面是否显示都会生效， 这一点和原版 AutoIt 的全局热键行为一致。

7. 数据文件格式

favlist.fav

描述#x#y#z
斯坦索姆-入口#1234.5#-678.9#12.3

描述里 - 前面的部分会被当作分类名，比如上面这行会归到"斯坦索姆"分类下。

hotkey.txt

# 以 # 开头的行是注释
^1
斯坦索姆-入口
!a
斯坦索姆-出口

两行一组：第一行是组合键（^=Ctrl，!=Alt，+=Shift），第二行是要跳转到的传送点描述（必须和 favlist.fav 里的描述完全一致）。

settings.json

首次运行会自动生成在 DLL 同目录下，记录分步移动参数、上次选择的分类、速度值等，缺失或损坏时会自动回退到默认值，不会导致崩溃。

8. 常见问题排查

  • 注入后没反应 / 按 Win+Insert 没有界面： 确认 DLL 真的被加载了（在 System Informer 里查看该进程的 Loaded Modules 列表里能不能找到 TeleportHackDX.dll）；确认游戏窗口正在用 D3D9 渲染（不是 D3D11/OpenGL 私服客户端）； 确认没被杀毒软件/Windows Defender 隔离（游戏外挂类工具很容易被误报，需要加白名单）。
  • 中文描述显示成方块/乱码： 程序运行时会自动从 C:\Windows\Fonts 加载微软雅黑/宋体等中文字体； 如果系统没有这些字体文件，会退回 ImGui 内置的纯 ASCII 字体，中文无法显示。
  • 点击 AntiJump/AutoLoot 等按钮提示 "unknown byte state"： 说明当前客户端在对应内存地址的字节内容和预期的两种已知状态（开/关）都不一致， 可能是客户端版本/补丁不完全匹配 src/domain/versions.cpp 里写死的偏移， 为了安全起见程序会拒绝写入而不是覆盖成未知内容。
  • Speed 按钮点了没反应： Speed 只在 3.3.5 客户端上支持，1.12.1/1.12.3 客户端本身没有这个可写字段。
